AI-native Typeless: A Best Practice for Voice Input in the Agent Era

Open Typeless Harness turns dictation into a feedback loop: listen, polish, insert into the focused field, watch corrections, and learn local speech skills.

TL;DR

GitHub: OpenCodexLabs/open-typeless-harness.

Open Typeless Harness starts from a simple frustration: speech-to-text keeps failing on personal vocabulary. Product names, paper terms, repo names, mixed Chinese-English phrases, and tiny style preferences are exactly the things generic dictation misses.

The best voice input loop is correction-native. It should watch what you fix after insertion, turn stable patterns into local speech skills, and make the next dictation better.

Why this exists

Voice input is usually treated as a one-shot conversion problem: speech goes in, text comes out. That is not how real writing works. Real writing includes correction, naming habits, mixed-language terms, project vocabulary, and context-specific phrasing.

For agent-era work, the input layer should adapt to the user's actual writing environment. If I dictate into a browser, an editor, a chat box, or a notes app, I should not have to move into a separate dictation workspace. I should speak where I already type, correct the result normally, and let those corrections become useful signal.

What the repo provides

The repository frames voice input as a small learning loop rather than a static transcript box.

StageWhat happensWhy it matters
ListenCapture speech and produce an ASR transcript.The user can stay in the current writing context.

InsertWrite text into the focused field.The app behaves like an input layer, not a separate editor.

AdaptPromote stable patterns into local speech skills.The next dictation can improve without manual prompt tuning.

The product value

The value is not that voice input becomes fancy. The value is that it stops repeating the same mistakes. If I always fix a project name, a paper term, or a Chinese-English phrase in the same way, the system should notice and adapt.

This turns voice input from a disposable transcript into a local learning surface. The user's corrections become a lightweight training signal for the next piece of writing.

How it fits AI-native workflows

AI-native work is not only about agents acting on files. It is also about changing how humans provide intent. Voice is a high-bandwidth intent channel, but only if the system can handle messy, personal, project-specific language.

Open Typeless Harness fits this gap. It keeps the user in the active app, lets the model clean the transcript, then learns from the exact corrections the user makes after the text lands.

When to use it

Use it whenBe careful when
You dictate technical or mixed-language content often.The target field contains private text you do not want monitored.
You repeatedly correct the same vocabulary mistakes.You need exact legal, medical, or financial transcription.
You want voice input inside existing apps.A simple keyboard shortcut is enough for the task.

Final recommendation

Voice input should not stop at transcription. The interesting product primitive is the correction loop: what did the user fix, which fixes repeat, and which ones should become local skills?

Open Typeless Harness is a best-practice experiment for that direction: focused-field dictation, LLM polish, post-insertion correction tracking, and local speech skills.
