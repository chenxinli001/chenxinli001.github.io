Agent Context Manager

Cross-harness context compression and recall for long-running agent work.

GitHub: OpenCodexLabs/agent-context-manager
DateMay 21, 2026
StatusProject note

TL;DR

GitHub: OpenCodexLabs/agent-context-manager.

Agent Context Manager is for people who do real work across more than one coding-agent window. The problem is familiar: you explained the constraints yesterday, debugged the failure in another harness, made a decision in a third session, and then the next agent starts from zero.

The core idea is simple: recall globally, compact locally, apply deliberately. It treats context as something that should be found, summarized, and reviewed, not something that should be silently shoved into a prompt.

Why this exists

AI-native work is becoming multi-session by default. A project may pass through Codex, Claude Code, CodeBuddy, Cursor, internal harnesses, browser traces, local notes, and command logs. Each tool may keep its own memory, but the user does not think in tool silos. The user thinks in projects, constraints, mistakes, and decisions.

That creates a new kind of friction. The valuable context is already somewhere on the machine, but it is not active at the moment when the next agent needs it. Re-explaining everything by hand is slow. Automatically injecting everything is dangerous. The missing product layer is context recovery with human review.

What the repo provides

The repository packages this idea as a small `ctx` workflow. It is not trying to become a full agent harness. It is a sidecar layer for context operations.

OperationWhat it meansWhy it matters
RecallSearch old local agent traces for relevant fragments.The next session can recover prior decisions without asking the user to restate everything.
CompactSummarize the current session into a smaller candidate context block.Long work can be carried forward without copying the whole transcript.
JobsQueue, preview, apply, or discard context results.Context stays reviewable before it affects the active agent.

The product value

The value is not that the agent has "more memory." More memory can make an agent worse if it is stale, irrelevant, or injected without inspection. The value is that the user gets a controllable working set.

For a coding agent, the active context should feel like a clean desk. The desk should contain the current goal, the important constraints, the decisions already made, the files already touched, and the open risks. It should not contain every random branch of conversation. Agent Context Manager is a practical way to rebuild that desk from previous work.

How it fits AI-native workflows

In an AI-native workflow, context is not a static note. It is an operating surface. The agent should be able to ask: where have we discussed this before, what was decided, what should I carry forward, and what should I ignore?

That is why review-first matters. The agent can propose a context block, but the user or the next agent can inspect it before applying it. This keeps the system useful without pretending that memory is always correct.

When to use it

Use it whenDo not overuse it when
You switch between multiple coding-agent tools.The task is a one-shot edit with no prior project context.
You need to recover old decisions, bugs, or constraints.The old trace may contain sensitive content you do not want searched.
You want compact handoffs between sessions.You need a fully automatic memory system with no human review.

Final recommendation

If agents are going to do serious multi-day work, they need a context recovery layer. Not because the model should remember everything, but because the user should not have to be the only bridge between yesterday's trace and today's task.

Agent Context Manager is a small best practice for that gap: search the old work, compact the current work, and apply context deliberately.
